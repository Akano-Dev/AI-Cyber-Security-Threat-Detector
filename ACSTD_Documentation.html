Section 01

Project Overview

ACSTD (AI Cyber Security Threat Detector) is a full-stack web application built as a security monitoring dashboard. It inspects incoming HTTP traffic, classifies requests as safe or malicious using a combination of rule-based detection and a trained machine learning model, and gives a security operator a live dashboard to monitor, block, or ignore threats in real time.

💡 Think of ACSTD as a security guard that reads every request before it reaches your website. If a request looks like an attack, it raises an alarm on the dashboard and blocks it. Safe requests pass through to the real site without the visitor noticing anything.

What Problem Does It Solve?

Most small websites and businesses cannot afford enterprise security tools. A single successful attack — SQL Injection, Cross-Site Scripting, or a brute-force login attempt — can expose user data, deface a website, or take down a service entirely. ACSTD provides:

  • Real-time detection of the most common web attack types
  • A live, visual dashboard that any non-technical team member can understand
  • One-click IP blocking that persists across server restarts
  • A reverse proxy mode that protects any existing website without code changes
  • Automatic brute-force detection and blocking

Section 05

Backend — How It Works

Startup Sequence

When you run python main.py, the following happens in order:

  1. Load .env file — reads TARGET_URL, RATE_LIMIT, RATE_WINDOW
  2. Initialize SQLite database — creates the threats and blocked_ips tables if they don't exist
  3. Load blocked IPs from the database into memory (a Python set for O(1) lookup speed)
  4. Load the trained ML model from model.pkl into memory
  5. Start the Uvicorn ASGI server on port 8000

Key Design Decisions

In-Memory Blocklist with Database Backup

Blocked IPs are stored in a Python set in memory so every request can check in O(1) time (instant lookup, no database query per request). When an IP is blocked, it is also written to the blocked_ips SQLite table so the block survives a server restart. On startup, the set is re-populated from the database.

Lifespan Context Manager

FastAPI's modern lifespan pattern (replacing the deprecated @app.on_event) is used to run startup logic once before the server begins accepting requests.

CORS Middleware

Cross-Origin Resource Sharing is configured to allow the React frontend at http://localhost:5173 to call the API. Without this, browsers would block the requests for security reasons.

Combined Verdict Logic

The detection pipeline combines two signals — signature rules and ML probability — using this logic: 

def _combined_verdict(sig_result, ml_prob):
    ml_pct = round(ml_prob * 100)

    # Signatures are authoritative (high precision, known patterns)
    if sig_result:
        confidence = max(ml_pct, 65) if ml_prob >= 0.5 else 65
        return sig_result["threat_type"], sig_result["severity"], confidence

    # ML-only: only flag if model is very confident (>= 70%)
    if ml_prob >= 0.70:
        return "Anomalous Payload", "medium", ml_pct

    return None, None, None  # safe

This means: if a signature matches, it's definitely a threat (the ML score just adjusts the confidence percentage). If no signature matches but the ML model is 70%+ sure, it's flagged as an "Anomalous Payload" (catches novel/unknown attacks). Anything below 70% ML confidence with no signature match is considered safe.

Section 06

Detection Engine (Rules + AI)

The detection engine in detection.py uses regular expressions (regex) — pattern-matching rules — to check if a payload contains known attack signatures. Think of it like a spam filter: it looks for specific keywords and patterns that are only ever seen in malicious requests.

Threat Signatures

Threat TypeSeverityWhat It DetectsExample Payload
SQL Injection High SQL keywords that try to manipulate database queries ' OR 1=1--, UNION SELECT, DROP TABLE
XSS High HTML/JavaScript tags injected into input fields <script>alert(1)</script>, onerror=
Path Traversal Medium Attempts to access files outside the web root ../../etc/passwd, ..%2F..%2F
Command Injection High Shell commands embedded in input to execute on the server ; ls -la, | cat /etc/passwd, `whoami`
Suspicious User-Agent Medium Known scanner/bot user agents used by attackers sqlmap, nikto, nmap, masscan
Brute Force High Automatic — triggered by rate limiter, not regex >20 requests in 60 seconds from one IP
⚠️ Signature rules have very high precision (low false positives) but miss novel attacks — attacks that don't match any known pattern. This is why the ML model is added as a second layer.

Section 07

Machine Learning Model

Overview

The ML model is a text classification pipeline trained to distinguish between malicious SQL injection payloads and normal text. It gives a probability score between 0 and 1 (0 = safe, 1 = malicious). This allows ACSTD to detect novel or obfuscated attacks that don't match any known regex signature.

Training Data

The model was trained on SQLiV3.csv, a real-world dataset from Kaggle containing approximately 30,919 rows. Each row is a web query labeled as either SQL injection or normal. The dataset contains real attack payloads collected from security research.

Pipeline

The model is a two-step scikit-learn pipeline:

Step 1 — TF-IDF Vectorizer (Feature Extraction)

Converts raw text payloads into numerical feature vectors. Configuration used:

  • analyzer="char_wb" — analyzes character-level n-grams (substrings), not whole words. This makes it robust to typos and obfuscation like SeLeCt instead of SELECT.
  • ngram_range=(2, 5) — looks at substrings of 2 to 5 characters long
  • max_features=8000 — keeps the 8,000 most important character patterns
  • sublinear_tf=True — dampens the effect of very frequent patterns

Step 2 — Logistic Regression (Classification)

A linear classifier that outputs a probability. Fast to train, fast to predict, and highly interpretable. Configuration:

  • C=5.0 — regularization strength (higher = less regularization, allows more complex boundary)
  • max_iter=1000 — allows enough iterations to converge on 30k rows

Section 08

Reverse Proxy System

What Is a Reverse Proxy?

A reverse proxy is a server that sits between users and your real web application. Users think they're talking to your app, but they're actually talking to ACSTD. ACSTD inspects the request and either blocks it or passes it through to the real app invisibly.

How the ACSTD Proxy Works

Any HTTP method (GET, POST, PUT, DELETE, etc.) to http://localhost:8000/proxy/<anything> is handled as follows:

  1. Check blocklist — if the source IP is in the blocklist, immediately return a styled 403 HTML page
  2. Check rate limit — if the IP has exceeded the request threshold, auto-block and return 429
  3. Build payload string — concatenates the URL path + query string + request body into one string for analysis
  4. Run detection — passes the payload through the signature engine and ML model
  5. If threat — logs to DB, broadcasts to dashboard via WebSocket, returns styled 403 page
  6. If safe — forwards the original request (headers + body) to TARGET_URL using httpx
  7. Return response — passes the real site's response back to the user unchanged

Configuration

Edit backend/.env to configure the proxy:

# The website to protect
TARGET_URL=https://your-site.com

# Block IPs that send more than 20 requests in 60 seconds
RATE_LIMIT=20
RATE_WINDOW=60

You can also change these live from the dashboard's Proxy Settings panel without restarting the server. Changes take effect immediately and are also written back to .env.

Section 09

Rate Limiting & Auto-Block

Brute force attacks (trying thousands of passwords, or hammering an endpoint) have a distinctive signature: many requests from the same IP in a short time. ACSTD detects and stops this automatically.

Sliding Window Algorithm

ACSTD uses a sliding window counter per IP address. For each IP, it keeps a queue (deque) of the timestamps of recent requests. On every new request:

  1. Remove all timestamps older than RATE_WINDOW seconds from the front of the queue
  2. Add the current timestamp to the end
  3. If the queue length exceeds RATE_LIMIT, the IP is rate-exceeded
def is_exceeded(self, ip):
    now = datetime.now(timezone.utc).timestamp()
    dq  = self._hits[ip]
    # Drop timestamps outside the window
    while dq and dq[0] < now - RATE_WINDOW:
        dq.popleft()
    dq.append(now)
    return len(dq) > RATE_LIMIT

What Happens When Rate Is Exceeded

  1. A Brute Force / Rate Abuse threat is logged (severity: HIGH, confidence: 99%)
  2. The IP is immediately added to the blocklist
  3. The block is persisted to SQLite — survives server restarts
  4. A WebSocket event is broadcast — the threat appears on the dashboard instantly
  5. All future requests from that IP get a 403 block page (not just 429)
🚫 Rate limiting counts are stored in memory only — they reset when the server restarts. This is intentional: a restart clears the count for all IPs, giving them a fresh start. Only actual blocks are persisted.

Section 12

Database Schema

SQLite is used with two tables. The database file is backend/threats.db and is created automatically on first run.

Table: threats

CREATE TABLE threats (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    timestamp   TEXT    NOT NULL,          -- "2024-01-15 10:30:00" (UTC)
    source_ip   TEXT    NOT NULL,          -- "192.168.1.100"
    threat_type TEXT    NOT NULL,          -- "SQL Injection", "XSS", etc.
    payload     TEXT    NOT NULL,          -- The raw malicious string
    severity    TEXT    NOT NULL           -- CHECK: 'low', 'medium', 'high'
                CHECK(severity IN ('low','medium','high')),
    status      TEXT    NOT NULL DEFAULT 'new'
                CHECK(status IN ('new','blocked','ignored')),
    confidence  REAL                       -- 0–100, from ML model
)

Table: blocked_ips

CREATE TABLE blocked_ips (
    ip         TEXT PRIMARY KEY,           -- "192.168.1.100"
    blocked_at TEXT NOT NULL               -- "2024-01-15 10:30:00" (UTC)
)

The blocked_ips table uses INSERT OR IGNORE — blocking the same IP twice doesn't cause an error, it just silently does nothing.

Section 15

Threat Types — Plain English Explanations

SQL Injection

Your website talks to a database to fetch user data, products, etc. It does this by sending SQL (database commands). SQL Injection is when an attacker tricks the website into including their own commands in those queries — e.g., ' OR 1=1-- makes a login query always return "true", bypassing the password check entirely. Worse variants can delete entire databases (DROP TABLE users) or extract all user data.

XSS (Cross-Site Scripting)

An attacker injects JavaScript code into a page that other users will see. When another user loads that page, the malicious script runs in their browser — it can steal session cookies (logging in as that user), redirect them to phishing pages, or silently send their data to the attacker's server.

Path Traversal

Web servers serve files from a specific folder. Path traversal uses sequences like ../../ to "climb up" the folder tree and access files outside the web root — such as /etc/passwd (Linux user list) or server configuration files containing passwords.

Command Injection

Some web apps pass user input directly to the server's operating system to run a command (e.g., to resize an image or send an email). An attacker can append their own shell commands: filename.jpg; rm -rf / — the server would process the image AND delete all its files.

Suspicious User-Agent

Every browser identifies itself with a "User-Agent" string (e.g., "Chrome/120"). Security scanners and automated attack tools like sqlmap, Nikto, and Nmap have distinctive user-agent strings that no real browser would send. Detecting these identifies automated reconnaissance — the step attackers take before a real attack.

Brute Force / Rate Abuse

Trying thousands of username/password combinations automatically. No clever hacking — just speed. ACSTD detects this by counting requests per IP over time. When an IP exceeds the threshold, it is auto-blocked before the attack can succeed.
